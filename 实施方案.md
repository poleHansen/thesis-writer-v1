# 融合式 AI PPT 系统实施方案

## 1. 建设目标

在现有分析基础上，目标不是简单拼接三个项目，而是构建一个新的统一系统，兼具以下能力：

1. 支持两类输入入口：

- 用户通过对话描述需求生成 PPT。
- 用户上传 Word、PDF、Markdown、网页链接等材料生成 PPT。

2. 具备专业 PPT 生产流程：
   - 先澄清需求。
   - 再做资料理解与必要研究。
   - 再产出提纲、页规划、视觉策略。
   - 最后生成高质量页面并导出 PPTX。
3. 兼顾产品化与最终交付质量：
   - 前台有 Web/API、项目管理、任务状态和模板管理。
   - 后台有稳定的内容处理中间层、规划层、SVG 渲染层和导出层。
4. 最终交付以 SVG-first 为核心，以保证页面精度、风格一致性和较好的 PPT 可编辑性。
5. 产品定位面向通用 PPT 生成网站，而不是单一论文写作工具；论文答辩只作为重点支持场景之一。

该系统本质上融合三类优势：

- Linux.do 帖子https://linux.do/t/topic/1782304?u=polehansen 的方法论优势：先问清楚、先研究、先规划。
- LandPPT 的平台优势：项目管理、研究工作流、模板中心、Web/API 化。
- ppt-master 的生成优势：严格的 SVG 模板体系、页面渲染流程、PPTX 导出链路。

## 2. 融合原则

### 2.1 方法层、平台层、渲染层分离

避免把“提问逻辑”“项目管理”“页面渲染”混在一起。系统拆成三层：

1. 方法层
   - 负责需求澄清、研究增强、提纲生成、页级规划、版式策略。
2. 平台层
   - 负责用户、项目、文件、任务、模板、预览、审阅、导出管理。
3. 渲染层
   - 负责把页规划转换成最终 SVG 页面并导出 PPTX。

### 2.2 双入口、单中台

无论是用户问答还是文件上传，最终都进入统一的内容理解与规划流程，避免做两套生成系统。

### 2.3 先内容定型，再视觉生成

先得到稳定的内容中间表示，再做视觉设计与渲染。这样更利于：

- 支持二次编辑。
- 支持重新换模板。
- 支持局部重生成。
- 支持人工审阅和流程回退。

```text
内容理解中间层
├─ 文档解析
├─ 内容清洗
├─ 结构提取
├─ 多模态资产分析
└─ Source Bundle 标准化

方法论引擎层
├─ 需求澄清 Agent
├─ 研究增强 Agent
├─ 提纲生成 Agent
├─ 页规划 Agent
└─ 视觉策略 Agent

设计与渲染层
├─ 模板选择器
├─ Design Spec Builder
├─ SVG Slide Renderer
├─ SVG Validator / Finalizer

├─ SVG -> PPTX
├─ HTML/PDF 预览导出
└─ 素材与结果归档
```

### 3.2 三套方案在目标架构中的角色映射

#### Linux.do 方法映射

放入方法论引擎层，重点保留：

- 先提问澄清目标、受众、用途、风格、页数。
- 对陌生主题先做研究，再写提纲。
- 提前产出 planning draft，而不是直接出最终页面。
- 在页级上使用 Bento Grid / 卡片化布局思维组织信息。

#### LandPPT 能力映射

放入接入与编排层、平台管理层，重点保留：

- Web 页面与 API 服务。
- Project、TODO、状态流转、文件管理。
- 模型路由、多 Provider 支持。
- 模板管理中心、研究报告管理、预览与任务追踪。

#### ppt-master 能力映射

放入设计与渲染层、导出层，重点保留：

- 基于 design_spec 的页面设计约束。
- 基于 SVG 模板的最终页面渲染。
- SVG 合规检查、finalize、PPTX 导出。
- 对 PPT 可编辑性的约束体系。

## 4. 核心数据流设计

### 4.1 统一输入模型：Source Bundle

系统不直接拿“原始输入”去画页面，而是先转换成统一中间对象 `Source Bundle`。

建议结构如下：

```json
{
  "project_id": "uuid",
  "input_mode": "chat | upload | hybrid",
  "user_intent": {
    "topic": "string",
    "goal": "string",
    "audience": "string",
    "scenario": "report | defense | pitch | training | product_launch | gov_briefing | tech_share",
    "constraints": {
      "pages": 12,
      "style": "consulting | academic | tech | govt",
      "language": "zh-CN"
    }
  },
  "sources": [
    {
      "type": "pdf | docx | md | url | text | image",
      "path": "string",
      "title": "string",
      "parsed_text": "string",
      "metadata": {}
    }
  ],
  "assets": {
    "images": [],
    "charts": [],
    "tables": []
  },
  "normalized_markdown": "string"
}
```

作用是把多种输入压平成统一内容载体，供后续提纲和页规划阶段复用。

### 4.2 内容规划模型：Presentation Brief Package

在 Source Bundle 之上，方法论引擎输出 `Presentation Brief Package`，它是“内容已基本定型，但还未进入最终渲染”的阶段性成果。

建议包含：

- 演示目标
- 目标受众
- 核心结论
- 章节结构
- 页面数量建议
- 页面主叙事线
- 风格建议
- 图表和配图策略
- 风险与不确定点
- 场景标签与交付语境，例如答辩、汇报、路演、培训、发布会等

### 4.3 页级执行模型：Slide Plan JSON

真正驱动渲染的不是原文，而是页级规划对象。

建议每一页包含：

```json
{
  "slide_id": "s01",
  "slide_type": "cover | toc | section | content | ending",
  "title": "string",
  "takeaway": "string",
  "layout_mode": "hero | bento | two_column | chart_focus | timeline",
  "content_blocks": [
    {
      "type": "paragraph | bullets | table | chart | image | quote | metric",
      "priority": 1,
      "content": {}
    }
  ],
  "visual_guidance": {
    "density": "low | medium | high",
    "highlight": "string",
    "icon_hint": "string",
    "image_hint": "string"
  },
  "speaker_notes": "string"
}
```

它是方法层和渲染层之间最关键的契约。

## 5. 核心流程设计

## 5.1 流程 A：用户对话生成 PPT

1. 用户输入主题或任务。
2. 系统进入需求澄清流程。
3. 生成初始 Brief。
4. 如主题信息不足，触发研究增强。
5. 生成提纲与页规划。
6. 用户确认或调整提纲。
7. 选择模板与风格。
8. 进入 SVG 渲染与导出。

这个流程主要继承 Linux.do 的方法论优势。

## 5.2 流程 B：用户上传文件生成 PPT

1. 用户上传 PDF、Word、Markdown、URL 或其他材料。
2. 系统执行解析与格式转换。
3. 将多来源内容合并为 `Source Bundle`。
4. 抽取核心结构、章节、图表、图片、数据源。
5. 生成 `Presentation Brief Package`。
6. 生成页规划。
7. 用户审阅提纲与模板建议。
8. 进入 SVG 渲染与导出。

这个流程主要继承 LandPPT 的文档接入能力，并以 ppt-master 完成最终高质量交付。

## 5.3 流程 C：混合模式

用户既上传文件，又通过对话补充要求，例如：

- “根据这份论文做成 15 页答辩 PPT。”
- “重点放大第三章实验结果。”
- “改成麦肯锡风格，适合给评审老师看。”

混合模式将成为主流模式，因此系统设计应以 `hybrid` 为一等公民，而不是附加逻辑。

## 6. 模块设计

### 6.1 接入层模块

#### 1. Web App

职责：

- 发起新项目。
- 上传文件。
- 发起对话式需求。
- 查看生成进度。
- 审核提纲、页规划和预览。
- 下载 PPTX / PDF / SVG。

#### 2. API Gateway

职责：

- 对外暴露统一接口。
- 管理鉴权、限流、项目路由。
- 统一收敛前端、异步任务和模型调用。

#### 3. Project Orchestrator

职责：

- 维护项目状态机。
- 管理任务编排。
- 串联解析、研究、提纲、渲染、导出各阶段。

建议状态：

- `created`
- `ingesting`
- `analyzing`
- `researching`
- `planning`
- `reviewing`
- `rendering`
- `finalizing`
- `exported`
- `failed`

### 6.2 内容理解层模块

#### 1. File Ingestion Service

负责接收并归档原始文件。

支持：

- PDF
- DOCX
- Markdown
- URL
- 图片
- 纯文本

#### 2. Document Parser Service

负责将不同文件统一转为结构化 Markdown 或结构化 JSON。

推荐策略：

- PDF -> Markdown / structured text
- DOCX -> Markdown
- URL -> cleaned markdown
- 图片 -> OCR + caption + layout extraction

#### 3. Content Normalizer

负责：

- 去噪。
- 去重复。
- 标题层级修复。
- 表格、图片、数据块识别。
- 合并多文件来源。

#### 4. Asset Analyzer

负责：

- 抽取图片。
- 检测图表与表格。
- 识别可复用视觉素材。
- 生成图片使用建议。

### 6.3 方法论引擎模块

#### 1. Requirement Clarifier

适用于对话入口或文件入口后的补充澄清。

负责问清：

- 给谁看
- 为什么做
- 想强调什么
- 页数要求
- 风格偏好
- 输出语言

#### 2. Research Enhancer

当输入内容不足时补足外部背景知识。

职责：

- 外部检索
- 生成研究摘要
- 输出可引用的背景材料

#### 3. Outline Generator

负责生成章节级结构。

输出：

- 一级目录
- 二级页面分配
- 每章核心结论

#### 4. Slide Planner

负责生成页级 `Slide Plan JSON`。

这是整个系统的关键控制点，因为页面质量很大程度取决于这一步，而不是最后渲染模型本身。

#### 5. Visual Strategy Generator

职责：

- 选择模板
- 生成设计参数
- 分配布局模式
- 指定图表、配图、图标策略

### 6.4 渲染与导出模块

#### 1. Template Registry

负责统一管理模板元数据。

建议模板元数据包含：

- 模板类型
- 风格标签
- 场景标签
- 信息密度
- 适配布局模式
- 可用页面类型
- design_spec 路径

#### 2. Design Spec Builder

将模板设计规范与项目需求组合成当前项目的最终设计上下文。

#### 3. SVG Slide Renderer

职责：

- 根据 `Slide Plan JSON` 和 `design_spec` 生成每页 SVG。
- 严格遵循 ppt-master 的 SVG 兼容约束。

#### 4. SVG Validator / Finalizer

职责：

- 检查非法 SVG 特性。
- 统一资源路径和页面结构。
- 为后续导出做收尾处理。

#### 5. Export Service

导出优先级建议：

1. 主链路：SVG -> PPTX
2. 辅链路：HTML 预览 -> PDF
3. 归档链路：SVG 源文件、notes、metadata

## 8. 各目录职责说明

### 8.1 apps/web

面向最终用户，提供：

- 新建项目
- 上传资料
- 对话补充要求
- 审阅提纲
- 审阅页规划
- 查看预览
- 导出结果

### 8.2 apps/api

作为平台总入口，管理：

- REST / WebSocket 接口
- 任务编排
- 项目状态管理
- 模型调用封装
- 文件与模板服务

### 8.3 packages/core-types

定义全系统共享数据契约，必须优先建设，避免后续各模块各自发明 JSON 结构。

优先定义：

- `SourceBundle`
- `PresentationBrief`
- `SlidePlan`
- `ProjectState`
- `TemplateMeta`

### 8.4 packages/ingestion

封装所有输入类型的解析逻辑，是支持“上传文件生成 PPT”的基础。

### 8.5 packages/methodology-engine

实现 Linux.do 帖子中最值得保留的专业流程逻辑，是整个系统的“脑”。

### 8.6 packages/template-registry

连接平台和渲染器：

- 平台负责挑模板。
- 渲染器需要知道模板的结构、页面类型和 design spec。

### 8.7 packages/svg-renderer

这是最接近 ppt-master 核心能力的模块，应严格控制技术边界，不要把业务流程逻辑塞进来。

### 8.8 packages/export-service

统一封装导出，避免导出逻辑散落在渲染器和 API 层。

## 9. 建议实施路径

### Phase 1: 统一数据模型

先定义：

- `SourceBundle`
- `PresentationBrief`
- `SlidePlan`
- `TemplateMeta`
- `ProjectState`

这是后续一切模块解耦的前提。

### Phase 2: 先做双入口与内容理解层

优先打通：

- 对话入口
- 文件上传入口
- PDF/DOCX/MD/URL 转 Markdown
- Source Bundle 生成

做到“输入统一”。

### Phase 3: 做方法论引擎

依次实现：

1. Requirement Clarifier
2. Outline Generator
3. Slide Planner
4. Research Enhancer
5. Visual Strategy Generator

做到“内容定型”。

### Phase 4: 对接 SVG 渲染内核

将 ppt-master 的关键能力抽成可调用服务：

- template resolution
- design_spec injection
- SVG rendering
- validate / finalize
- PPTX export

做到“高质量交付”。

### Phase 5: 补全平台能力

逐步补强：

- 项目状态机
- 异步任务队列
- 版本管理
- 审阅反馈
- 局部重生成
- 导出管理

做到“可运营”。

## 10. 第一阶段最小可用版本建议

为了避免系统过大，MVP 建议只保留最关键链路：

1. 支持两种输入：
   - 输入主题
   - 上传 PDF / DOCX / MD
2. 输出三类中间结果：
   - Brief
   - Outline
   - Slide Plan
3. 支持选择一个模板族。
4. 输出 SVG 预览。
5. 输出 PPTX。

MVP 暂时可以不做：

- 复杂协作权限
- 多人评论
- 大型模板市场
- 超细粒度页面编辑器

## 11. 关键技术决策建议

### 11.1 页面最终表示采用 SVG-first

原因：

- 更接近 ppt-master 的可编辑 PPT 目标。
- 更容易实现高一致性的视觉输出。
- 比 HTML 截图式导出更适合追求“像专业设计稿”的结果。

### 11.2 内容中间层采用 Markdown + Structured JSON 双表示

原因：

- Markdown 适合承接多格式文档解析。
- Structured JSON 适合提纲、页规划、模板映射与渲染执行。

### 11.3 模板体系采用“元数据 + design_spec + 页面骨架 SVG”

这样既能继承 ppt-master 的模板质量，又能具备 LandPPT 风格的模板中心能力。

### 11.4 平台任务采用异步编排

因为以下步骤耗时明显：

- 文档解析
- 外部研究
- 图片生成
- SVG 渲染
- 导出 PPTX

必须用异步任务和状态回传，而不能做同步阻塞接口。

## 12. 结论

这套融合方案的核心不是“把三个项目堆在一起”，而是明确三者分工：

- Linux.do 负责方法论。
- LandPPT 负责平台化。
- ppt-master 负责最终页面质量和导出质量。

最终形成的系统应是：

1. 双入口统一接入。
2. 单一内容中间层统一承载。
3. 方法论引擎先定内容与结构。
4. SVG 渲染器再定最终视觉与导出。
5. 平台层负责项目、任务、模板、审阅和结果管理。
