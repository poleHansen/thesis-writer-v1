# 融合式 AI PPT 生成网站分阶段实施计划

## 1. 文档定位

这份文档不是高层架构说明，而是面向实际落地的执行计划。目标是把已经确定的 AI PPT 生成网站架构拆成可以逐步开发、逐步验收、逐步上线的阶段任务。

适用前提：

1. 系统目标已经确定，且目标产品是融合 Linux.do、LandPPT、ppt-master 优势的 AI PPT 生成网站。
2. 总体架构已经在 [实施方案.md](d:/code/thesis-writer-v1/实施方案.md) 中定义。
3. 当前需要一份团队可直接照着推进的开发计划。

本计划默认采用以下总体策略：

1. 先统一数据模型，再统一输入。
2. 先打通内容规划链路，再接入 SVG 渲染。
3. 先做最小可用产品，再做平台化增强。
4. 所有阶段都要求可验证、可回退、可交付。

## 2. 总实施原则

### 2.1 先搭骨架，再做能力

第一优先级不是“赶快生成某一类学术 PPT”，而是先把项目结构、数据契约、状态流转和任务编排框架定住，使系统能支撑多场景 PPT 生成。否则后面所有模块会越来越耦合。

### 2.2 统一中间对象优先于模型提示词优化

当前系统成败不取决于提示词写得多花，而取决于以下 3 个对象是否稳定：

1. `SourceBundle`
2. `PresentationBrief`
3. `SlidePlan`

这三者必须先定义清楚，再去写 Agent 和渲染器。

### 2.3 每阶段结束必须有可演示成果

每一阶段都要求形成一个清晰的“可看见结果”，例如：

- 能上传文件并解析出标准化内容。
- 能生成适用于答辩、汇报、路演、培训等场景的提纲。
- 能生成页规划 JSON。
- 能生成 SVG 页面。
- 能导出 PPTX。

### 2.4 避免一次性集成所有复杂能力

以下能力放到后期，不要在前期同时推进：

- 多人协作评论
- 高级在线编辑器
- 模板商店
- 复杂权限矩阵
- 全量多模型自动对比

## 3. 推荐总工期视角

如果按一个小团队执行，建议拆成 8 个阶段：

1. 阶段 0：项目启动与技术基线
2. 阶段 1：核心数据模型与项目骨架
3. 阶段 2：双入口与内容理解层
4. 阶段 3：方法论引擎第一版
5. 阶段 4：模板系统与 SVG 渲染内核
6. 阶段 5：导出链路与结果管理
7. 阶段 6：平台化增强与审阅闭环
8. 阶段 7：稳定性、质量与上线准备

建议节奏：

- 阶段 0 到阶段 2：打底层，优先级最高。
- 阶段 3 到阶段 5：打通核心价值链。
- 阶段 6 到阶段 7：做产品化和稳定性。

## 17. 结论

这份计划的核心思想是：

1. 不要一开始就追求“大而全”。
2. 先把双入口和统一中间层做扎实。
3. 再把方法论引擎做出来。
4. 同时把 Web 入口逐步收敛成真正可用的产品工作台。
5. 最后用 SVG-first 渲染和导出把质量拉高。

只要严格按本计划推进，团队就不会陷入“每个模块都像在做，但整体一直打不通”的典型陷阱。整套系统会按照清晰依赖顺序逐步成型，并且每个阶段结束后都有真实可验证的交付结果。
